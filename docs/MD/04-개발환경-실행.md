# 개발환경 & 실행 가이드

## Docker Compose (권장 배포)

```powershell
cd Y:\Project\destiny\DIM_Wishlist_hub
docker compose build                 # 코드 변경 반영 (프론트는 --no-cache frontend 권장 — 함정 4)
docker compose up -d                 # 앱 http://localhost:8080, 백엔드 docs http://localhost:8000/docs
# 실 데이터 적재 (매니페스트 버전 동일 시 --force 필요)
docker compose run --rm ingest python -m ingest.manifest_ingest --force
docker compose restart backend
```
- published 포트는 **127.0.0.1 전용**(LAN 미노출 — 보안). LAN 공유하려면 인증 추가 후 바인딩 변경.
- nginx 가 `/api/*` 를 backend 로 프록시(동일 출처). 업로드 한도 `client_max_body_size 16m`.
- `.env`(BUNGIE_API_KEY 등)는 `env_file` 로 읽힘. 데이터는 `dimdata` 볼륨에 영속.

## 실행 방법 (로컬 개발, Docker 없이)

### 백엔드 (FastAPI)
```powershell
cd Y:\Project\destiny\DIM_Wishlist_hub\backend
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
uvicorn app.main:app --reload      # http://127.0.0.1:8000/docs
```
Bungie API 키 없이도 `seed_data.json` 의 샘플 무기 4종으로 즉시 구동된다.

### 프론트엔드 (이 머신: WebDAV 우회 필수)
이 프로젝트는 WebDAV 가상 드라이브(`Y:`)에 있어 `cd frontend && npm run dev` 가 실패한다(아래 함정 1).
**래퍼 스크립트로 실행**한다:
```powershell
cd Y:\Project\destiny\DIM_Wishlist_hub
node scripts/dev-local.mjs          # http://localhost:5173
```
- 프론트엔드를 로컬 디스크(`%LOCALAPPDATA%\dimhub-fe`)로 동기화한 뒤 거기서 Vite 를 띄운다.
- 소스/설정은 매 실행 동기화 → **코드 수정 후에는 스크립트를 재시작**해야 반영된다(라이브 HMR 은
  로컬 복사본 기준).
- Claude 미리보기 패널의 `frontend` 구성도 이 스크립트를 사용하므로 패널에서 바로 열린다.
- 일반 로컬 디스크(WebDAV 아님)에 클론한 경우엔 그냥 `cd frontend && npm run dev` 로 충분.

프론트엔드는 `/api/*` 요청을 백엔드(`:8000`)로 프록시하므로 **백엔드가 먼저 떠 있어야** 데이터가 보인다.

## 실제 데이터 적재 (선택)
1. https://www.bungie.net/en/Application 에서 API Key 발급.
2. 루트 `.env`(`.env.example` 복사)에 `BUNGIE_API_KEY=...`, `MANIFEST_LOCALE=ko`.
3. 적재:
   ```powershell
   $env:PYTHONUTF8=1
   backend\.venv\Scripts\python.exe -m ingest.manifest_ingest
   backend\.venv\Scripts\python.exe -m ingest.voltron_bootstrap
   ```
적재 후 백엔드를 재시작하면 `data/app.sqlite` 를 자동 감지해 상태가 `매니페스트`로 바뀐다.

## 이 머신 특유의 함정 3가지 (검증 중 확인)

### 1) WebDAV 가상 드라이브(`Y:`) — Vite dev/build 실패
- `Y:` 는 RaiDrive WebDAV(`\\RaiDrive-USER\WebDAV`). Vite 의 esbuild/rollup(Go 기반 경로
  해석)이 `Y:` 를 접근 불가한 `Z:` 로 정규화 → `ENOENT ... Z:\...` 로 실패.
- 백엔드(파이썬)와 Node 직접 실행은 `Y:` 에서 정상. **번들러만** 문제.
- 해결: `scripts/dev-local.mjs` 로 로컬 디스크에서 실행(위 참조). `preserveSymlinks` 로는 해결 안 됨.

### 2) npm 전역 설정 `os=linux`
- `npm config get os` → `linux`. 윈도우에서 리눅스 플랫폼 바이너리를 설치해 Vite 가
  `@rollup/rollup-win32-x64-msvc` / `@esbuild/win32-x64` 를 못 찾는다.
- 해결: 설치 후 명시 추가
  `npm install --no-save @rollup/rollup-win32-x64-msvc @esbuild/win32-x64`
  (또는 `npm config delete os`). 버전 핀 없이 plain 설치가 동작.

### 3) Python 3.8 **32-bit**
- `requirements.txt` 는 `uvicorn[standard]` 대신 **plain `uvicorn`** 사용. `[standard]` 의
  httptools/uvloop 는 win32 cp38 휠이 없어 MSVC 컴파일이 필요(실패). pydantic v2 는 휠이 있어 정상.

### 4) dev-local 동기화 시 `Copy-Item` 디렉터리 중첩
- `dev-local.mjs`/수동 동기화로 `frontend/src` 를 `%LOCALAPPDATA%\dimhub-fe\src` 로 복사할 때,
  대상 `src` 가 이미 있으면 `Copy-Item -Recurse` 가 **`dimhub-fe\src\src` 로 중첩** 복사해 Vite 가
  구버전을 서빙한다. 해결: 복사 전 `Remove-Item "$dst\src" -Recurse -Force` 후 복사.
- **도커 프론트는 무관**(빌드 컨텍스트가 원본 `Y:\...\frontend`). 단, WebDAV mtime 캐시로 `npm run build`
  레이어가 잘못 캐시될 수 있어 **`docker compose build --no-cache frontend`** 권장.

## 자주 겪는 증상 → 원인
- **프론트엔드가 "로딩"에서 멈춤**: 포트 5173 을 점유한 깨진 Vite(=WebDAV 에서 기동) 가 마운트 안
  되는 페이지를 서빙. → 5173 점유 프로세스 종료 후 `node scripts/dev-local.mjs` 로 재기동.
- **데이터가 안 보임**: 백엔드(:8000) 미기동. → 백엔드 먼저 실행.
- **`npm run dev` 가 `Z:\...` ENOENT**: 함정 1. → 래퍼 스크립트 사용.
- **폰트가 늦게 적용**: Google Fonts 비동기 로드(렌더 비차단). 시스템 폰트로 즉시 렌더 후 교체 — 정상.
