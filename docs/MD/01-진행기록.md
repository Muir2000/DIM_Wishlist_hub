## 2026-06-27 — 외부 공개 HTTPS 배포 (NPM) + `apply_schema` 마이그레이션 버그 수정

NAS(UGREEN DXP4800 Plus) 구동 앱을 **Nginx Proxy Manager + Let's Encrypt** 로 `https://muir20.ddns.net`
외부 공개. 구성: 공유기 80→41537/443→41391 포워딩, NPM 프록시(Scheme `http` → `192.168.0.101:8080`,
Force SSL/HTTP2), OAuth 도메인 대응(`.env` 의 `BUNGIE_OAUTH_REDIRECT_URI`/`FRONTEND_URL` 을 도메인으로,
`SESSION_COOKIE_SECURE=1` 추가) + Bungie 포털 Redirect URL 갱신. 상세 → [07-외부공개-HTTPS배포.md](07-외부공개-HTTPS배포.md).

**버그 수정 — `db.apply_schema()` 실행 순서**: `SCHEMA` 내 `CREATE INDEX idx_profiles_owner ON
scoring_profiles(owner)` 가 레거시 DB(owner 컬럼 부재)에서 `executescript` 중 먼저 실행되어
`OperationalError: no such column: owner` → 백엔드 부팅 실패(재시작 루프). 인덱스 생성을 `SCHEMA` 에서
빼 **ALTER(컬럼 보강) 이후**로 이동([backend/app/db.py](../../backend/app/db.py)). 코드가 이미지에 빌드되므로
`docker compose up -d --build backend` 로 반영. 또한 백엔드 재생성 시 IP 변경 → 프론트 nginx 가
옛 IP 캐싱(정적 `proxy_pass`)으로 `/api` 502 → **프론트엔드 컨테이너 재시작**으로 해소.

> ⚠ 외부 공개는 [05-보안검증.md](05-보안검증.md) 의 단일 사용자 전제와 충돌 — 멀티유저/인증 전제 충족 시에만 유지할 것.

## 2026-06-25 — 점수 비중(스코프 블렌드)·열 세분화·기대치 환산 (v4)

**스코프 비중(블렌드).** 퍽 가중치를 "가장 강한 스코프 하나"에서 **동일 무기(0.60)·프레임(0.25)·종류(0.15)
절대 가중합**으로 변경(`scoring._perk_weight`). 같은 plug_hash 공유 덕에 무기별 데이터가 없어도 같은
종류·프레임에서 평가가 전이된다.

**열 세분화 + 동적 만점.** 총열/탄창/특성/기원 열 기여 차등(특성 1.0·총열탄창 0.35·기원 0.2·intrinsic 0),
**열별 최선 퍽** 채택, 점수 = 100×(열가중 기여+조합)/**무기별 만점**(채점가능 열 비중 합) → "풀롤 대비 %"
기대치. 일반화↓·천장↑(특성 2개만 ≈74, 풀롤 100).

**미등록 무기 캡(coverage).** 무기별 위시리스트가 없으면 무기 비중(0.60)을 비워 **coverage(종류+프레임=0.40)
로 점수 캡** → 새 무기 판단 보조. 빌더에 "종류·프레임 기준 — 최대 40%" 안내, 툴팁에 열 비중 환산 기여 표시.
임계 재보정 god 75/viable 40. 프로필에 `scope_blend`·`column_weights` 필드(기본값+JSON 조정).
`POST /score`(+coverage·max_possible), `POST /score/perk-weights`(+max_possible·coverage·column_weights·kinds).
점수 테스트 23개로 확장(스코프 블렌드·열 비중·동적 만점·coverage 캡·combo 캡), 백엔드 74/74.

## 2026-06-24 — 시즌(복각) 라벨링 + 외부 위시리스트 표기 조사

**배경.** 같은 이름 무기라도 시즌(복각)마다 퍽 풀이 다르고, 같은 시즌의 HOLOFOIL 은 동일 퍽 풀을
공유한다(검증: 폭발 용광로 = S5 원본 40·39·28퍽 중 시즌별 상이, 홀로포일은 같은 시즌 일반판과 풀
100% 일치). 외부 위시리스트(`wish_list_test.txt`, MrCharles Aegis)를 조사한 결과, 커뮤니티는
**시즌을 명시하지 않고** 같은 이름 무기의 변형 해시를 `// 이름` 블록으로 나열하며, 노트에
`Origin: <기원특성>` 으로 출처를 구분할 뿐이다. 우리 도구는 "어느 시즌/몇 번째 시즌" 표기가 약했음.

**조사 결론.**
- 무기 정의에는 `seasonHash` 가 비어 있어 무기→시즌을 직접 알 수 없다.
- 유일한 시즌 식별자는 `iconWatermark`(시즌마다 바뀜). watermark→시즌번호 매핑은 매니페스트에
  없고 DIM 커뮤니티(`d2-additional-info/output/watermark-to-season.json`, 80개·최대 S28)가 관리.
- 시즌명은 매니페스트 `DestinySeasonDefinition`(번호+이름)에서 가져온다.

**적용.**
- `backend/app/refdata/` 에 `watermark-to-season.json`(DIM) + `season-names.json`(매니페스트 ko/en,
  ASCII-안전 JSON) 동봉. Docker 이미지에 포함(`.dockerignore` 미제외 확인).
- 신규 `backend/app/seasons.py`: 런타임 1회 로드 → `season_for_watermark(watermark) → (번호, 이름)`.
  절대 URL/상대 경로 모두 처리. **재적재 불필요**(watermark 만으로 매 요청 매핑).
- `models.WeaponSummary` 에 `season_number`/`season_name` 추가, `serialize.weapon_summary` 가
  원본 watermark 로 매핑해 부착.
- 검색을 **시즌(변형 그룹+watermark) 단위**로 1개씩 접어 표시(이전 변경) → 각 행에 시즌 칩
  `S{번호} · {시즌명}` + 워터마크 아이콘 + `복각 N시즌` 태그. 같은 시즌 홀로포일은 한 행으로 병합.
- 무기 헤더(빌더)에도 `시즌 N · 시즌명` + 워터마크 표시.
- 위시리스트 내보내기는 기존대로 굴릴 수 있는 모든 시즌 해시에 자동 전개(per-variant 풀 필터).

**검증(도커 :8080).** 폭발 용광로 → 3행: `S28 · 업적의 기념비` / `S23 · 소원의 시즌` /
`S5 · 대장간 시즌`, 각 시즌 풀 상이(돌격 탄창은 S5 원본엔 없음). 프론트 시즌 칩 렌더 확인.

> ⚠️ `watermark-to-season.json` 은 DIM 스냅샷(최대 S28). 신규 시즌 출시 시 갱신 필요
